# 서론
이 문서는 GitHub Pages로 **주피터 노트북(`main.ipynb`)을 Jupyter Lab 스타일 HTML로 변환**해 주차별 URL로 배포하는 표준 절차를 정리한 가이드입니다.  
`algorithm_lv4`, `algorithm_practice_lv1` 등 동일한 주차 디렉터리 구조를 쓰는 저장소에 공통으로 적용할 수 있습니다.

목표는 다음과 같습니다.
- 단일 HTML이 아닌 `week{N}/` URL 구조를 유지한다.
- 브라우저에서 노트북 본문(마크다운·코드·Mermaid 등)이 **nbconvert 결과물**과 같이 읽기 좋게 표시된다.
- 이후 다른 작업자가 같은 절차를 반복해도 동일한 결과를 얻는다.

# 핵심 요약
- Pages **소스 브랜치**는 `gh-pages`, **경로**는 저장소 루트(`/`)를 사용한다.
- 루트 `index.html`은 **목차(주차 링크)** 용도로 둔다.
- 주차별 **실제 페이지**는 `week{N}/index.html`이며, 내용은 **`week{N}/main.ipynb`를 `jupyter nbconvert`로 HTML 변환**한 결과를 그대로 둔다(파일명만 `index.html`).
- 저장소 루트에 **`.nojekyll`** 파일을 두어 Jekyll이 `_`로 시작하는 경로 등을 건드리지 않게 한다.
- `gh-pages`에 **`index.md`와 `index.html`을 동시에 두지 않는다.** Jekyll이 마크다운을 우선하면 화면이 기대와 달라진다.
- `main.py`는 배포 HTML의 본문이 아니다. 노트북에 이미 코드가 있으면 별도 링크는 선택 사항이다.
- 반영 후 **`gh-pages`만 push**하면 배포가 갱신된다(빌드 완료까지 1~2분 걸릴 수 있음).

# 본론

## 디렉터리 구조(gh-pages)

```text
gh-pages
├── .nojekyll
├── index.html
├── week1/
│   └── index.html    ← main.ipynb → nbconvert 결과
├── week2/
│   └── index.html
└── weekN/
    └── index.html
```

## 운영 URL 예시

- 루트 목차: `https://<계정>.github.io/<저장소>/`
- 주차 페이지: `https://<계정>.github.io/<저장소>/week4/`

참고: 동일 방식으로 배포된 예시는 [algorithm_practice_lv1 week11](https://jongphago.github.io/algorithm_practice_lv1/week11/) 과 같다.

## 사전 요구 사항

- `jupyter nbconvert` 사용 가능(예: `jupyter nbconvert --version` 확인).
- 로컬에 `week{N}/main.ipynb`가 최신 상태인지 확인한다.

## 신규 주차 또는 갱신 배포 절차

아래는 `week4`를 예로 든다. 번호만 바꿔 반복하면 된다.

```bash
# 1) gh-pages 브랜치로 이동 (원격과 맞추려면 pull)
git checkout gh-pages
git pull origin gh-pages

# 2) (선택) main 등 작업 브랜치의 노트북을 가져와 반영하는 경우
#    예: git checkout main -- week4/main.ipynb
#    그 후 다시 gh-pages로 돌아와 변환한다.

# 3) 노트북 → HTML 변환 (출력 파일명 main.html)
jupyter nbconvert --to html "week4/main.ipynb" --output "main.html"

# 4) GitHub Pages 규칙에 맞게 주차 진입 파일로 복사
cp week4/main.html week4/index.html
rm -f week4/main.html

# 5) 루트 index.html에 week4 링크가 없으면 추가한다.

# 6) .nojekyll이 없으면 루트에 빈 파일로 추가한다.
#    touch .nojekyll

# 7) 커밋 및 푸시
git add week4/index.html index.html .nojekyll
git commit -m "week4 노트북 HTML 배포 갱신"
git push origin gh-pages

# 8) 작업 브랜치 복귀(선택)
git checkout main
```

## GitHub 저장소 설정

- 저장소 Settings → Pages → **Branch**: `gh-pages`, **Folder**: `/ (root)`.

## 안전 수칙

- `gh-pages`에서 대량 삭제 명령(`git rm -rf .`)은 가급적 사용하지 않는다.
- 기존 주차 페이지를 유지한 채 필요한 `week{N}/`만 추가·수정한다.
- 배포 직후 Pages 상태가 `building`이면 1~2분 후 Actions 또는 API로 다시 확인한다.
- **Markdown 인덱스 재도입 금지**: 루트·주차 모두 `index.html`만 두고 노트북 HTML로 통일하는 것을 권장한다.

## main.html을 미리 두는 저장소의 경우

일부 저장소는 `main` 브랜치에 `week{N}/main.html`을 커밋해 두고, 배포 시에만 복사한다.

```bash
git checkout gh-pages
mkdir -p week12
git show main:week12/main.html > week12/index.html
# 이후 index.html 목차 갱신, commit, push
```

`algorithm_lv4`는 **소스가 `main.ipynb`인 경우 위 nbconvert 절차**를 기본으로 한다.

# 결론
Pages 배포의 핵심은 **`gh-pages`에서 `week{N}/index.html`이 노트북과 동일한 읽기 경험을 주는 HTML**이라는 점이다.  
`jupyter nbconvert`로 생성하고, `.nojekyll`과 **HTML 전용 인덱스**만 유지하면 [algorithm_practice_lv1 week11](https://jongphago.github.io/algorithm_practice_lv1/week11/) 과 같은 렌더링을 재현할 수 있다.
